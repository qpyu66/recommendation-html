전체 흐름
📦
원시 데이터 수집
IN: 서비스 로그/카탈로그 → OUT: 정규화된 이벤트 테이블
›

추천 시스템의 입력은 3가지 데이터입니다. 이벤트 로그에서 어떤 행동을 얼마나 강한 선호 신호로 볼지 가중치를 정의하는 것이 출발점입니다.

IN
서비스에서 쌓인 원본 데이터
  • events.csv: 누가, 무엇을, 언제, 어떻게 반응했는지
  • users.csv: 유저 식별자와 기본 속성
  • items.csv: 상품의 카테고리, 브랜드, 태그, 가격
처리
추천에 필요한 신호만 남김
  • 행동 종류를 view, click, purchase로 구분
  • 행동별 중요도를 EVENT_WEIGHTS로 정의
  • 최근 행동은 크게, 오래된 행동은 작게 보도록 기준 설정
OUT
모델이 읽을 수 있는 이벤트 테이블
  • user_id, item_id, event_type, timestamp
  • 각 이벤트가 선호 신호인지 계산할 준비 완료
다음 사용처
피처 엔지니어링

다음 단계에서 이벤트마다 점수를 만들고, 유저별 선호 프로필로 집계합니다.

# events.csv
user_id,item_id,event_type,timestamp
u1,i1,view,2026-06-01T09:00:00
u1,i1,click,2026-06-01T09:01:00

# config.py
EVENT_WEIGHTS = {
    "view":     1.0,  # 봤다
    "click":    3.0,  # 관심 있다
    "purchase": 8.0,  # 강한 선호
}
TIME_DECAY_HALF_LIFE_DAYS = 14
↓
⚙️
피처 엔지니어링
IN: 이벤트 테이블 → OUT: 유저/아이템 피처 프로필
›

원시 이벤트를 유저 선호 프로필로 변환합니다. 행동 가중치 × 시간에 따른 영향 감소로 각 이벤트의 신호 강도를 계산하고, 이를 카테고리·브랜드·태그·가격대 선호로 집계합니다.

IN
정규화된 이벤트 + 상품 메타데이터
  • event_type: 행동 강도 계산에 사용
  • timestamp: 오래된 행동의 영향 감소에 사용
  • items.csv: 카테고리, 브랜드, 태그, 가격 매핑
처리
행동을 숫자 피처로 변환
  • 행동 가중치 × 최근성 점수로 이벤트 강도 산출
  • 유저가 반응한 상품 속성을 누적
  • 카테고리, 브랜드, 태그, 가격대를 정규화
OUT
유저 선호 프로필
  • categories: 선호 카테고리 분포
  • brands, tags: 선호 속성 분포
  • price_norm: 선호 가격대
다음 사용처
CBF/CF/Hybrid 모델 추론

CBF는 유저 프로필과 아이템 프로필의 유사도를 계산하고, CF는 유저 행동 벡터를 비교합니다.

# features.py
def event_strength(event, reference_time):
    base  = EVENT_WEIGHTS[event["event_type"]]
    days  = (reference_time - event["timestamp"]).days
    decay = 0.5 ** (days / TIME_DECAY_HALF_LIFE_DAYS)
    return base * decay

# 결과: 유저 선호 프로필
{
  "categories": {"shoes": 0.71, "shirt": 0.29},
  "brands":     {"nike": 0.62, "adidas": 0.38},
  "tags":       {"running": 0.48, "casual": 0.31},
  "price_norm": 0.42
}
↓
🧠
모델 추론
IN: 피처 프로필/행동 벡터 → OUT: item별 추천 점수
›

CBF는 아이템 메타데이터 유사도로, CF는 비슷한 유저의 행동으로 후보를 생성합니다. Hybrid는 두 점수를 가중합산합니다.

IN
모델별로 필요한 피처
  • CBF: 유저 프로필 + 아이템 프로필
  • CF: 유저-아이템 행동 행렬
  • Hybrid: CBF 점수 + CF 점수
처리
후보 아이템에 점수 부여
  • CBF: 속성 유사도를 가중합산
  • CF: 비슷한 유저가 좋아한 아이템을 찾음
  • Hybrid: 두 점수를 하나의 최종 점수로 합침
OUT
추천 후보 리스트
  • item_id: 추천할 상품
  • score: 높을수록 우선 노출
  • model: cbf, cf, hybrid 중 어떤 방식인지
다음 사용처
아티팩트 저장

유저와 모델별 추천 리스트를 JSON으로 저장해 API가 빠르게 읽을 수 있게 합니다.

# CBF (models/cbf.py) — 메타데이터 유사도
score = (0.35 * category_score
       + 0.15 * brand_score
       + 0.35 * tag_score
       + 0.15 * price_score)

# CF (models/cf.py) — 비슷한 유저 행동 기반
similarity = cosine_similarity(
    my_vector, neighbor_vector
)

# Hybrid (models/hybrid.py)
CBF_WEIGHT, CF_WEIGHT = 0.55, 0.45
final = CBF_WEIGHT * cbf_score \
      + CF_WEIGHT  * cf_score

파이프라인 실행 · 아티팩트 저장
IN: 추천 결과/프로필 → OUT: artifacts/*.json
›

pipeline.py를 실행하면 모든 유저 × 모든 모델의 추천 결과가 미리 계산되어 JSON으로 저장됩니다. 서빙 시 실시간 추론 없이 이 파일을 즉시 읽어 응답합니다.

IN
배치 계산 결과
  • 유저별 선호 프로필
  • 아이템별 피처 프로필
  • 모델별 추천 후보와 점수
처리
서빙용 파일로 고정
  • 모든 유저 × 모델 조합을 미리 계산
  • API가 읽기 쉬운 JSON 구조로 저장
  • 추천이 부족할 때 쓸 인기 아이템도 준비
OUT
아티팩트 파일
  • recommendations.json: 최종 추천 리스트
  • user_profiles.json, item_profiles.json: 설명/디버깅용 프로필
  • popular_items.json: fallback용 인기 상품
다음 사용처
서빙 API

API 서버가 이 파일들을 메모리에 올린 뒤 요청마다 조회만 수행합니다.

# 실행
python3 -m src.pipeline

# 생성 파일
artifacts/
  recommendations.json  # {cbf/cf/hybrid: {u1: [...]}}

  item_profiles.json    # 아이템별 피처
  popular_items.json    # fallback용 인기 아이템
↓
🚀
서빙 API
IN: user_id/model/k 요청 → OUT: 추천 API 응답
›

serve.py는 아티팩트를 메모리에 로드하고 요청마다 즉시 조회합니다. 결과가 k개 미만이면 인기 아이템으로 자동 fallback합니다.

IN
클라이언트 요청
  • user_id: 추천을 받을 유저
  • model: cbf, cf, hybrid 중 선택
  • k: 몇 개를 받을지
처리
미리 만든 결과 조회
  • recommendations.json에서 유저/모델 키 조회
  • 요청한 개수만큼 잘라서 반환
  • 결과가 부족하면 인기 아이템으로 채움
OUT
화면에서 바로 쓸 응답
  • items[]: item_id와 score 목록
  • fallback_used: fallback 사용 여부
  • model: 어떤 추천 로직의 결과인지
사용처

모델 로직 영역 · DS 주도
CBF/CF/Hybrid 점수 계산은 모델 내부 판단입니다. 어떤 알고리즘을 쓰고 어떤 유사도를 더 신뢰할지는 DS가 주도하는 영역입니다. 개발팀은 내부 공식을 결정하기보다 입력, 출력, latency, artifact 형식을 이해하는 쪽에 가깝습니다.
src/pipeline.py · artifacts/*.json
운영 인수 경계 · DS 산출물이 서비스 산출물로 바뀌는 지점
pipeline.py는 모델 결과를 recommendations.json, user_profiles.json, item_profiles.json, popular_items.json으로 고정합니다. 개발팀이 서빙에서 인수하는 것은 DS의 노트북이나 모델 내부 코드가 아니라, 재현 가능하고 검증 가능한 artifact입니다.
src/serve.py
서빙 영역 · 개발팀 주도
API 요청 처리, fallback, 응답 형식, 서버 상태 확인은 서비스 안정성 영역입니다. 모델 품질 판단은 DS와 협업하되, 운영 중 장애 대응, latency, rollback, 모니터링 기준은 개발팀 책임에 가깝습니다.

핵심 문장
모델 소유권과 운영 인수는 다르다

DS가 모델 로직을 소유하더라도, 개발팀이 책임질 수 있는 것은 운영 가능한 형태로 전달된 artifact입니다. 계약 없는 실험 결과는 바로 서비스 책임으로 넘어오지 않습니다.
